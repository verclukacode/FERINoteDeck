# Backend Overview

NoteDeck's backend is a Node.js + Express REST API written in TypeScript, persisting data to MySQL via Prisma, and using Firebase Authentication for identity.

## Authentication flow

Every request (except `/api-docs` and static image serving) passes through `requireAuth`:

1. Frontend gets a **Firebase ID token** from `firebase.currentUser.getIdToken()`.
2. Token is sent as `Authorization: Bearer <token>`.
3. `requireAuth` calls `admin.auth().verifyIdToken(token)`.
4. On success it **upserts** a `User` row in MySQL (`id = firebaseUid`, `email`, optional `username`).
5. `req.user` is set to `{ uid, email }` — all route handlers scope queries to `req.user.uid`.

New users must choose a username (`username = null` after first sign-in); the frontend redirects to `/choose-username` until this is set.

## Data model

```
User ──< Folder ──< Page ──< NoteInvite
     ──< FlashcardFolder ──< Deck ──< FlashCard ──< Review
     ──1 StudySettings
     ──< CalendarTag ──< CalendarEvent
     ──< CalendarEvent (tagId nullable)
```

### Key entities

**User** — `id` is the Firebase UID. `username` starts null and must be set before the app is usable.

**Folder / Page** — Standard notes hierarchy. `Page.content` is a markdown string with a `<<<NoteDeckMD>>>` sentinel (custom block-editor format). `Page.isPublic` + `publicDescription` + `publishedAt` drive the marketplace.

**FlashcardFolder / Deck / FlashCard** — Flashcard hierarchy. `FlashCard` holds all SRS scheduling state: `state` (`new`/`learning`/`review`/`relearning`), `due` (unix ms BigInt), `intervalSec`, `ease` (permille), `reps`, `lapses`, `learningStep`.

**Review** — Immutable revlog entry written on every `POST /api/cards/:id/answer`. Stores before/after state for analytics.

**StudySettings** — Per-user Anki-style config: `newCardsPerDay`, `maxReviewsPerDay`, learning/relearning step sequences, ease multipliers, interval modifier.

**NoteInvite** — Lets a user share a note (`Page`) with another user by username. Status: `pending` → `accepted` / `declined`.

**CalendarTag** — User-defined label with a `name` and one of the 6 folder colors (`red`, `orange`, `pink`, `green`, `blue`, `purple`). Tags are scoped per user. Deleting a tag sets `tagId` to `null` on affected events (`SetNull` cascade).

**CalendarEvent** — An exam, test, or any future deadline. Fields: `name`, optional `description`, `date` (DateTime), optional `tagId`. Used by `GET /api/calendar-events/upcoming` to drive urgency toasts.

## SRS scheduler (`src/lib/srs.ts`)

Implements an **Anki-style SM-2** algorithm:

- Cards have four states: `new` → `learning` → `review` (or `relearning` after a lapse).
- Grades: `1 Again`, `2 Hard`, `3 Good`, `4 Easy`.
- All times are **unix milliseconds stored as `BigInt`**; intervals are **seconds as `BigInt`**; ease is **permille** (2500 = 2.5×).
- JSON serialization: `BigInt` fields go through `src/lib/serialize.ts` (converts to `string`) before responses are sent.
